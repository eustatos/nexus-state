## Обоснование

### Почему это правильно

1. **Типы не выполняются** — TypeScript типы стираются при компиляции в JavaScript
2. **Нет runtime кода** — файлы содержат только интерфейсы, type aliases, константы
3. **Искусственное завышение** — 100% покрытие типов не означает качество кода
4. **Стандарт индустрии** — большинство TypeScript проектов исключают файлы типов
5. **Фокус на логике** — coverage отражает тестирование бизнес-логики, не объявлений

### Что проверяет TypeScript

- ✅ Корректность типов — `tsc --noEmit`
- ✅ Совместимость типов — компилятор
- ✅ Экспорты/импорты — модульная система
- ✅ Константы — компилятор проверяет типы

### Что проверяют тесты

- ✅ Runtime логика — функциональность
- ✅ Бизнес-правила — поведение
- ✅ Обработка ошибок — edge cases
- ✅ Интеграции — взаимодействие компонентов

## Преимущества

1. **Более точный coverage** — отражает только тестируемый runtime код
2. **Честные метрики** — нет искусственного завышения от типов
3. **Стандарт индустрии** — как в большинстве TypeScript проектов
4. **Фокус на качестве** — coverage мотивирует тестировать логику, не типы
5. **Меньше шума** — отчёты о покрытии чище и понятнее
